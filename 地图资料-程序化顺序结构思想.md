## 顺序结构基本概念
顺序结构是自然存在的一种客观事实，在线性系统中，按时间先后执行的流程我们认为就是顺序结构。

## 程序逻辑：顺序执行流
```c
int main(void) {
  printf("hello world");
  printf("\n");
  printf("by ssks");
  return 0;
}
```
我们看经典的 C 语言 Hello World。这个函数首先在终端中打出`hello world`这一串鹰文，然后打出`\n`换行，最后在下一行打出`by ssks`又一串。
这样的「首先……然后……最后……」可以表明，上述`main(void)`当中的语句是逐条、自上而下执行的，这就是程序中的顺序结构。

既然如此，我们不妨想象“脑死”这个战役，也是按顺序执行的：先建锅盖，然后打`YAPPET`，然后秋风扫落叶。不妨把一局战役也类比成超大的`main`函数。

## 关键实现：允许、禁止触发
但我们注意到，有的任务开头会显示难度，同时播放经典 EVA，顺便把你鼠标锁了，等等。如此想来，触发并没有那么顺序。那是什么促使触发按我们期望的顺序走呢？答案是“允许触发”。

最近刚学会 Py 的协程，不妨就用`async`解释吧。正常来说，触发是同时在一个 loop 里（或者，类似优先队列？）等待执行的：
```python
await asyncio.gather(trigger1(), trigger2(), ...)
```
但忽然，触发 1 明确要求等触发 2，以此类推，形成等待链：
```python
async def t1():
  await t2()

async def t2():
  await t3()

asyncio.run(asyncio.gather(t1(), t2(), t3(), ...))
```
如此一来，`run`进去之后首先异步等待`gather`；然后 gather 取出不用等的`t1()`，执行中途发现要等`t2()`，t2 中途发现要等`t3()`，……  
一直到最后一个触发跑完，被等待的触发依次结束，`gather`跑完`run`执行结束，整个战役就通关了。

当然这个例子非常不恰当——
- t1 允许 t2 **实际上并不会把 t1 挂起来等**；
- 这个例子实际上把异步当成同步来用，完全没有发挥异步的优势。当然这不是重点。

举协程的例子是想说明，我们“允许触发”人为控制了触发的时序，保证了触发按照我们想的顺序执行。这样的触发设计就叫顺序结构设计，并且这种设计最广泛地为 mapper 所使用。
