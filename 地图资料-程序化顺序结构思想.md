## 顺序结构

顺序结构是自然存在的一种客观事实，在线性系统中，按时间先后执行的流程我们认为就是顺序结构。

---

顺序结构是最简单、最基本、最符合思维直觉的结构。在程序代码当中，顺序是通过代码行的先后次序来体现的：

```python
x = input("输入 x：")
y = input("输入 y：")
print(x + y)
```

可以很清楚地看出，这段程序先读入`x`，后读入`y`，最后输出它们两相加的结果。**自上而下、逐行执行**，这就是程序代码的顺序结构。

同样地，触发乃至一整关的战役，最常见的流程结构也是顺序结构。以被广为改编的「脑死」为例，它的任务流程具有很典型的顺序性：首先需要建立一座~~锅盖~~苏军雷达，然后打掉最后一座心灵控制器，最后秋风扫落叶。

在触发里要保证顺序，一个简单的办法是用行为*53 - 允许触发*。假定有两个触发 $t_0$ 和 $t_1$，要求先触发 $t_0$ 后触发 $t_1$。只需在触发编辑器里完成如下步骤：
- 为 $t_1$ 触发勾上「禁止」选项；（$t_0$ 无需再做处理）
- 在 $t_0$ 触发的行为（或者结果）页面中，添加一项（行为类型选中 53，参数选择 $t_1$ 触发）。

如需链式允许一系列触发，比如 $t_1 \to t_2 \to \ldots \to t_n$ ，也是如法炮制。以此类推，就形成触发链：

---

我们知道，触发具有“若 p 则 q”的语义。从**逻辑**上，可以把它看作一个`if`单分支：

```python
if elapsedTime(6):  # 13 - 流逝时间
    text_trigger("mission:naosi_A")  # 11 - 文本触发
```

那么上述的顺序结构也可以在逻辑上用程序代码表示（即伪代码）。不妨把这些`if`也按照地图师期望的顺序，\*自上而下\*地排列起来：

> [!note]
>
> 我不太清楚 GitHub MarkDown 是否允许编写 LaTeX 伪代码，至少我博客使用的`katex`插件不可以。伪代码姑且还是用 Python 表示，但不会引入难懂的语言特性和语法机制。

```python
# t0. start
if anyEvent:  # 8 - 任何事件（当*单独使用*时，它会令触发*立即执行*）
  lock_input()  # 46 - 禁止用户输入
  play_speech("EVA_EstablishBattlefieldControl")  # 21 - 播放 EVA 语音

  # t1. intro.brief.A
  if elapsedTime(6):  # 13 - 流逝时间
    text_trigger("mission:naosi_A")  # 11 - 文本触发
```
其中，$t_0$ 触发的条件是`anyEvent`。结合右边的注解和触发 $p \to q$ 的语义，这个条件肯定是恒满足的，也就是为真（`True`）。

上述代码段的排布与前面的案例一致，$t_1$ 是排在 $t_0$ 后面的，这样 $t_1$ 必定会等 $t_0$ 先触发；除此之外，$t_1$ 还往右缩进，与 $t_0$ 的那两条结果对齐，意味着 $t_0$ 不触发时，绝对不会触发 $t_1$，以此避免独立触发和触发链的歧义。
